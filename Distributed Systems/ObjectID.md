# ObjectID

MongoDB's 12-byte primary key format that encodes a timestamp, node identity, and sequence counter into a compact, roughly time-sortable identifier generated client-side.

## Why it matters

MongoDB needs IDs that can be generated by any driver instance (not by the server) without coordination, while still being roughly ordered by insertion time and carrying enough entropy to avoid collisions across a cluster.

## How it works

12 bytes (96 bits), displayed as 24 hex characters, with three components:

| Bytes | Bits | Content |
|-------|------|---------|
| 0–3   | 32   | Unix timestamp (seconds) — big-endian, enables sorting |
| 4–8   | 40   | Random value seeded once per process (encodes machine + process identity) |
| 9–11  | 24   | Incrementing counter, initialized to a random value at startup |

The timestamp component is 4 bytes (seconds-level), not milliseconds. Two ObjectIDs generated within the same second on different machines may not sort correctly relative to each other — they're only reliably sortable at second granularity.

Because the counter and random value are per-process, ObjectIDs generated by the same driver process within a second are strictly ordered; across processes they're ordered only by second.

## Key tradeoffs

- **Timestamp prefix** — gives useful sort order and index locality (avoids the B-tree scatter of random [UUIDs](UUID.md)), but only at second granularity
- **Client-side generation** — no server roundtrip for ID assignment; at the cost of requiring drivers to implement the spec correctly
- **96 bits vs. 128** — smaller than a UUID but larger than a [Snowflake ID](Snowflake%20ID.md); the timestamp field isn't adjustable, so the format expires in 2106
- **Not globally unique forever** — the 40-bit random value is re-seeded per process start, not guaranteed globally unique across all machines at all times; collision risk is negligible in practice

## Related concepts

- [Document Databases](../Databases/Document%20Databases.md) — ObjectID is MongoDB's native `_id` type; MongoDB drivers generate it automatically when no `_id` is provided
- [UUID](UUID.md) — the 128-bit alternative; v7 UUIDs improve on ObjectID's timestamp resolution with millisecond precision
- [Snowflake ID](Snowflake%20ID.md) — a 64-bit alternative with stricter ordering guarantees and millisecond resolution
- [Sharding](../Databases/Sharding.md) — the timestamp prefix means ObjectIDs have good write locality in range-partitioned indexes but can still create a write hot spot on the most recent shard
- [Partitioning](../Databases/Partitioning.md) — the timestamp prefix benefits range-partitioned tables by keeping recent writes co-located in the current partition
